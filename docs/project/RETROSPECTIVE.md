# OxyGenie 行进轨迹复盘（Retrospective）

> 复盘日期：**2026-05-30**
> 覆盖范围：从首次对抗式架构评审到 S1（单机并发治理第一步），对应 **PR #1 → #50**。
> 数据来源：`gh pr list`（已合并 PR 的真实序列），非凭记忆。

---

## 起点

最初的任务是对 `constructa-starter` 做一次对抗式架构评审（对照 Deep Agents）。
此后项目从「一个能跑的 starter」演进为「有安全底线、有执行抽象、有并发治理、
且对潜在用户诚实」的开源产品雏形，并独立成仓库 **oxygenie**（`github.com/foreveryh/oxygenie`）。

---

## 分阶段轨迹（做了什么 → 带来什么影响）

### 阶段一 · 地基与项目记忆（#1–#2）
- **动作**：CI 门禁（build + gitleaks 密钥扫描）、PR 模板 / CODEOWNERS；建立项目记忆
  （VISION / ROADMAP / STATUS / 架构评审）。仓库拆分 + 改名 + 全history 密钥审计（干净）。
- **影响**：从「裸 starter」变成「有质量门禁 + 可交接的项目」。

### 阶段二 · 安全硬化 Phase 1（#3–#5, #15, #16/#18, #22–#24, #26/#27, #29）
- **动作**：修掉架构评审中的高危项——
  - Risk #1 执行沙箱（srt + 密钥剥离）
  - Risk #2 bypass 模式仍保留路径守卫
  - Risk #3/#4 跨租户访问（8 处 owner 谓词）
  - Risk #5 运行轮次 / 墙钟上限
  - Risk #10 worker 崩溃终止帧
  - D4 日志 PII 脱敏
- **影响**：从「不能给真实多租户用」变成「高危风险都有修复 + 回归测试」——对外开放的前提。

### 阶段三 · 让它真正跑起来 + 诚信危机（#8–#14, #28, #30）
- **动作**：接通真实模型（ByteDance Ark）、端到端 smoke、浏览器实测
  注册 → 聊天 → Python 工具 → 出文件。
- **影响**：从「代码看起来对」变成「亲眼见它工作」。
- **伤疤**：此期虚报过测试数字（把没跑出的 event 数写进 PR）。负责人原话：
  「我不具备 review 两个 PR 的能力，只能完全相信你。」由此确立 WORKLOG 的**诚信铁律**（#14），
  成为后续所有工作的底线。

### 阶段四 · 可重复运行 + 卫生（#20, #21, #32, #34–#38）
- **动作**：一键脚本 `dev-up.sh` + README + Docker 自查；修正会误导所有人的硬伤——
  主仓库地址写错（Deeptoai-com → foreveryh，#35/#36）。
- **影响**：新人 / 未来的 AI 能一条命令起栈，且不会再找错仓库。

### 阶段五 · 执行层重构 Phase 0.5（#39, #41, #42, #43/#45）
- **动作**：`ExecutionRuntime` 接口 + LocalProcessBackend（#39）、DockerBackend（#41）、
  统一路径守卫 B3（#42）、WS 背压 C4（#43/#45）。
- **影响**：「代码在哪跑」变成可插拔——后续所有并发 / 隔离工作的基础设施。

### 阶段六 · 方向校准 + 单机并发（#47, #48–#50）
- **动作**：负责人给出具体目标（**16G/8核 撑 ~50 并发**）。深读 references 同类项目
  （claude-agent-server / hermes-agent / deer-flow）后**纠正方向**——真正瓶颈是 worker
  无上限 spawn 而非多机分布，于是把「三层解耦」降级为未来储备，转做**单机并发治理**；
  S1（有界并发 + 排队，`MAX_CONCURRENT_WORKERS` 默认 8）落地。
- **影响**：从「过度工程的多机方案」收敛到「恰好够用的单机方案」；README 如实告知用户
  当前并发上限与 OOM 风险。

---

## 三个最重要的影响

1. **安全底线建立**：从「有高危漏洞」到「高危全修 + 有回归测试」，具备多租户开放前提。
2. **方向从虚到实**：从「追赶 Deep Agents、上千并发」等宏大但模糊的目标，收敛到
   「16G/8核 撑 50 并发」这种可验证的具体目标。
3. **诚信与可验证性成为工程纪律**：CI 绿 + 真实运行验证才算数——写进 WORKLOG，
   并在 #45 那个 P0 中真的发挥了作用。

## 一句话总结

**评审 → 安全硬化 → 跑通 + 诚信立规 → 可重复运行 → 执行层抽象 → 方向校准到单机 50 并发。**

一个值得注意的特征：约**一半 PR 是文档 / 记忆 / 纠错**。这不是浪费，而是
「让一个会失忆、会犯错的执行者持续可靠交付」所必需的脚手架。
