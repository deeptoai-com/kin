# 设计篇 04：评测 —— 黄金集、检索/生成指标、离线 + 在线、回归门禁

> baby-agent 把"LLM 评测"列为完整 Agent 的必备一章。oxygenie **完全没有**——没有 `eval/`、没有黄金集、没有 RAGAS。前面每一篇设计（RAG、记忆、上下文）改完都会问"更好了吗？"，没有评测就只能靠拍脑袋。评测是所有其他设计篇的前置。

> 📐 本篇原为**设计篇**（2026-06-07，现状有据，设计为"该有的"）。
> ✅ **2026-06-13 实现回填**：做 RAG（设计篇 01）时，**这套评测被一起做出来了**——黄金集、分层检索指标、消融模式都已落地，还实打实推翻了一个设计决定（rerank）。但只做了一半：**生成层指标和 CI 门禁仍未做**。详见末尾 [实现回填](#实现回填2026-06-13评测落地了一半还驱动了一次反悔)。

## 问题陈述

改了切分策略、换了 embedding、调了 rerank、动了 system prompt——**怎么知道是变好还是变坏？** 没有可重复的度量，每次"优化"都是赌博，且无法防回归。

## oxygenie 现状

- **零评测基建**：grep `eval/golden/ragas` 无果；无黄金数据集、无评测脚本、无基准套件。
- CI（第 18 篇）有 lint/typecheck/unit test 硬闸，但**没有 Agent/RAG 质量评测**这一关。
- 唯一的"在线信号"是 `usage_record`/`audit_log`（第 17 篇），但没被用来算质量。

## 朴素方案为什么不行

- **肉眼测**：人工试几条 case——抓不住回归，改 A 修好、悄悄弄坏 B 看不见。
- **只看在线指标**：等真实用户反馈太慢，且无法在上线前拦截。
- **只测最终回答**：RAG 出错可能在召回、在精排、在生成任一环，只测端到端定位不了是哪段坏了。

## 核心方案：离线黄金集 + 分层指标 + CI 门禁

- **黄金集**：按知识库维护 Q→相关 chunk / 期望回答 的标注集（可由团队策展，呼应北极星"精选而非公开市场"）。
- **分层指标**：检索层（Recall@K / MRR / NDCG / context-relevance）+ 生成层（faithfulness 忠实度 / answer-relevance / 是否带正确引用）——**分层才能定位是召回坏了还是生成坏了**。
- **离线 + 在线**：离线黄金集进 **CI**（第 18 篇）做回归门禁，关键指标跌破阈值就拦合并；在线从 `usage_record`/反馈（第 17 篇）采质量信号补充长尾。
- **落点**：新增 `eval/` 目录 + 黄金集；评测脚本可在 CI 跑；在线信号复用既有审计/用量表。

## 反直觉结论

> [!IMPORTANT]
> **评测不是"做完功能后的验收"，是"做功能的前提"。** 没有可重复的度量，设计篇 01-03 的每一次"优化"都无法证伪——你不知道换 embedding 是真的更好还是只是手感更好。把评测放到最前面、进 CI 门禁，才让后面所有改动**可证伪、可回归防护**。这也是为什么它排在设计篇靠前：**它是其他设计篇"是否更好"这个问题的唯一答案来源。**

## 三个生产坑

> [!WARNING]
> **坑一**：黄金集会过时——知识库一更新，旧标注就失真，要有维护节奏。
> **坑二**：LLM-as-judge（用模型打分 faithfulness）本身有方差，要固定评测模型 + 多次取均值，否则指标自己在抖。
> **坑三**：只测端到端会掩盖分层问题——必须能分别报"召回坏了"还是"生成坏了"。

## 实现回填（2026-06-13）：评测落地了一半，还驱动了一次反悔

设计篇说"评测是其他设计篇的前置"。这话在 RAG 落地时被验证了——**不是先建评测再做 RAG，而是做 RAG 的过程中被逼着把评测建起来**：换 embedding（doubao vs zhipu）、调切块、要不要开 rerank……每一个都得有数才能定。于是黄金集 + 分层指标真做了出来，并且**当场推翻了 D1 设计里"rerank 是管线一环"的假设**。

### 设计 ✅ 兑现的（且比设计更细）

**黄金集——而且按"该谁赢"分了型。** `tests/golden/rag-golden-set.ts`（~40 题）没有像设计只说的"Q→相关 chunk"那么粗，而是**按检索腿分类型**标注（`rag-golden-set.ts:5-7`）：
- `keyword`：目标里有原词 —— **BM25 的主场**
- `paraphrase`：与目标零字面重叠 —— **embedding 的主场**
- `entity`：罕见专名/代码 —— 混合检索的 tie-breaker
- `clause`：条款号题（如"第 3 条"）

**为什么这么分**：只用改写题（paraphrase）测，会得出"BM25 这条腿没用"的错误结论——因为改写题本就是向量的主场。给每条腿配它的主场题，才量得准它到底值不值得留。**这是设计篇"分层才能定位"的一个更锋利的版本：不只分检索/生成层，还分检索内部的每条腿。**

**分层指标 + 消融模式。** `scripts/rag-eval-v2.ts` 报 **R@1 / R@4 / R@8 + MRR**（`:84`），且按三种模式各跑一遍（`:26-29`）：`vector`（关 BM25+rerank）/ `hybrid`（关 rerank）/ `+rerank`。**这就是消融**——单独开关每条腿，看指标怎么动，精确定位"是哪条腿在起作用"。再按题型拆报（`:89-90`），一眼看出"BM25 在 keyword 题上救了多少分"。

### 设计驱动的一次真实反悔：rerank 默认关

这套评测最大的产出不是一张报表，是**一个被数据推翻的设计决定**。D1 设计图里 rerank 是"向量∥BM25→RRF→**rerank**→返回"的固定一环。`rag-eval-v2.ts` 一跑，真相出来了：生产 `k=8` 下，开不开 rerank **R@8 都是 96%**，rerank 只动前 4 名排序，但模型 8 条全读、收益被吃掉，却白付 +1.4 秒/查。于是 rerank 被设成默认关（`search.ts:56`）。

> [!IMPORTANT]
> **这正是设计篇反直觉结论"评测让优化可证伪"的兑现瞬间。** 没有这套 eval，"加了 rerank 应该更好"会作为一个想当然的设计一直留在管线里，白烧每一次查询的 1.4 秒。**评测的价值不是证明你对，是让你能发现自己错——并且是在上线前、用可重复的数字发现。**

### 仍是设计的一半（诚实标注）

- **❌ 生成层指标没做。** 设计说的检索层（R@K/MRR/NDCG）做了；**生成层（faithfulness 忠实度 / answer-relevance / LLM-as-judge）一个没建**。现在只能量"召回对不对"，量不了"答得忠不忠实于召回"。这也意味着设计「坑二（LLM-judge 方差）」还没机会踩到。
- **❌ CI 回归门禁没做。** eval 是 `scripts/` 下**手动跑**的脚本，没进 `.github/workflows`——`grep rag-eval .github/` 为空。设计最看重的"进 CI、跌破阈值拦合并"还差这一步。现状是"有尺子、但要人记得拿出来量"，离"自动量、量不过不许合"还有一截。

一句话：**RAG 把评测从"零"推到了"检索层可证伪 + 驱动了一次反悔"，但生成层和 CI 门禁这两块——评测真正成为"防回归硬闸"的关键——仍停在设计。** 黄金集会过时（设计坑一），目前还靠人维护节奏。
