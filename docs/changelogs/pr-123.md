# PR #123 — ci(changedoc): drop the auto-PR changelog-persistence steps

## TL;DR
- Removes the two CI steps that staged `docs/changelogs/pr-<N>.md` and opened a `chore/changelog/pr-<N>` PR per source PR.
- ChangeDoc output now lives **only** as a PR comment — the existing `edit-mode: replace` post-comment step is unchanged.
- Eliminates failures from the repo setting "Allow GitHub Actions to create and approve pull requests" being off.
- Cuts noise: no more auto-opened `changelog, automated` PRs per source PR.
- Replaces the deleted block with an inline NOTE comment recording the removal date (2026-06-07) and rationale.

## Why
The persistence flow tried to commit each ChangeDoc to `docs/changelogs/` via `peter-evans/create-pull-request`, but the repo disallows Actions from opening PRs, so the step red-X'd on every run. Even when it worked, it produced a churn PR per source PR. The PR comment already carries the useful signal, so persistence-via-auto-PR is dropped.

## What changed

### Build / CI
- `.github/workflows/ai-pr-docs.yml` — deleted two steps from the `changedoc` job:
  - **"Stage PR changelog file"** — shell step that created/updated `docs/changelogs/pr-<N>.md`, preserving the header above `## TL;DR` on re-runs.
  - **"Open/Update PR that persists the ChangeDoc"** — `peter-evans/create-pull-request@v7` step that pushed to `chore/changelog/pr-<N>` with labels `changelog, automated`.
- Left in place: the prior step that posts/replaces the ChangeDoc as a PR comment from `CHANGELOG_PR.md` (`edit-mode: replace`, `body-path: CHANGELOG_PR.md`), and the subsequent "Generate AI code review with Qwen Code" step.
- Added an inline NOTE comment in the workflow documenting the 2026-06-07 removal and rationale (blocked-by-setting + noise).

### API / UI / DB / Types / Tests
- None. Workflow-only change; no application code, schema, query keys, or types touched.

### Docs
- No docs files edited. Existing `docs/changelogs/pr-*.md` files (if any) are untouched; they simply won't get new siblings from this workflow.

## Breaking changes
- **CI behavior change, not a code break.** Downstream consumers relying on a `chore/changelog/pr-<N>` branch/PR or on `docs/changelogs/pr-<N>.md` being created per source PR will no longer see them. Switch to the PR comment, or restore persistence via a different mechanism (see follow-ups).

## Test plan
- Open a throwaway PR against `main` and confirm:
  - The ChangeDoc job runs green (previously red-X'd on the `create-pull-request` step).
  - The PR comment is posted via `edit-mode: replace`.
  - No `chore/changelog/pr-<N>` branch is created and no secondary PR is opened.
  - No `docs/changelogs/pr-<N>.md` file appears in the tree.
- Re-run the workflow on the same PR and confirm the comment is replaced (not duplicated) with no branch/PR side effects.

## Suggested follow-ups
- If a durable archive of ChangeDocs is wanted, persist them out-of-repo: upload `CHANGELOG_PR.md` as a workflow artifact, push to a docs site, or write to a separate repo via a PAT — rather than re-enabling the in-repo auto-PR.
- Alternatively, flip the "Allow GitHub Actions to create and approve pull requests" setting on and reinstate a slimmed-down version that only commits on merge to `main` (not on every PR sync).
- Audit existing `docs/changelogs/pr-*.md` files and decide whether to keep, archive, or delete them now that the directory is no longer maintained automatically.
- Consider removing the `changelog, automated` labels from the repo if nothing else uses them.
